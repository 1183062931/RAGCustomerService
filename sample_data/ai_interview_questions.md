# AI 常用面试题口语化回答稿

## 1. 什么是 RAG，为什么很多企业项目不用纯大模型直答？

如果面试官这么问，我一般会比较口语化地回答：

RAG 其实就是“先查资料，再让模型回答”。它的核心价值不是让模型更聪明，而是让回答更贴近企业自己的真实数据。纯大模型直答的问题在于，它虽然通用能力强，但不知道你们公司的产品文档、售后 SOP、合同条款、知识库更新内容，所以它很容易一本正经地答错。

我在生产里更常见的做法是，用户问题先走检索，把最相关的知识片段取回来，再和问题一起交给模型生成答案。这样做的好处有三个：第一，答案更可控；第二，能引用来源；第三，知识更新不需要重新训练模型。

比如做智能客服时，用户问“退款多久能到账”，如果系统只靠通用模型，它可能会说 3 到 7 天，但你们公司实际规则可能是“原路退款 1 到 3 个工作日，钱包退款实时到账”。这种场景就非常适合 RAG，因为答案必须基于企业当前规则，而不是模型自己的常识。

## 2. Embedding 是什么，它在 RAG 里起什么作用？

Embedding 你可以理解成“把文本变成向量表示”。这不是简单编码，而是把语义信息压缩到一组数字里。这样做的目的是，让“退款怎么处理”和“申请退货后多久能回款”这种字面不同但语义接近的句子，在向量空间里距离更近。

在 RAG 里，Embedding 最主要的作用就是做语义检索。我们会把知识库文档切成 chunk，每个 chunk 先转成向量存起来。用户提问的时候，也把问题转成向量，再去找最相近的内容。

我在实际项目里踩过一个坑，就是只关注模型选型，不关注 embedding 的领域匹配。比如一个制造业售后系统里，文档大量出现“模组、治具、老化测试、工位切换”这类词，如果 embedding 模型对这些术语理解一般，召回质量会明显下降。后来我们会结合实际问答日志抽样验证，看是不是出现“表面像相关，实际不解决问题”的召回结果。

## 3. 向量数据库为什么要用 Chroma、Milvus、FAISS 这类组件？

因为做语义检索，不能每次都拿用户问题和全部文本逐条比，这样性能会很差。向量数据库的价值就是把向量的存储、索引、近似搜索、元数据过滤这些事都做好。

如果我口语化解释，我会说：它就像传统数据库解决结构化查询的问题，向量数据库解决的是“按语义找相似内容”的问题。

生产环境里，我会根据场景选型。像单机、小规模、本地开发、PoC 阶段，用 Chroma 很合适，接入快，维护成本低。比如你这个客服系统，先用 Chroma 落地完全合理。等以后数据量上到百万级 chunk，或者要多副本、高并发、权限隔离，再考虑更重的方案。

## 4. 文档切分为什么重要？chunk 太大或太小会有什么问题？

文档切分其实直接决定 RAG 的上限。chunk 太大，检索结果里会混很多无关内容，模型读起来噪音大；chunk 太小，又会把完整语义切碎，导致检索回来的是半句话，模型没法稳定回答。

我一般会说，chunk 不是固定数字，而是“信息完整性”和“检索精度”之间的平衡。实际项目里，我会优先按标题、段落、列表、表格这种天然结构切，再叠加适当的 overlap。

比如做客服知识库时，一份“退换货规则”文档里同时包含退款时效、特殊商品、运费责任。如果整页塞成一个 chunk，用户问“生鲜能不能退”时，检索到的内容会把一堆退款流程也带出来。后来我们按二级标题和规则列表切分，命中质量会明显更好。

## 5. 什么是幻觉？你在项目里怎么降低幻觉？

幻觉本质上就是模型输出了看起来很合理、但实际上没有依据甚至是错误的内容。这个问题在客服、医疗、金融、法务这些场景尤其危险。

我在项目里不会只靠 prompt 去压幻觉，而是多层一起做：

第一层是检索约束，尽量让模型只基于命中的知识片段回答。  
第二层是提示词约束，明确告诉模型“找不到就说不知道”。  
第三层是输出后处理，比如检查是否带引用、是否超出已知知识范围。  
第四层是兜底策略，比如低置信度时转人工。

举个生产例子，有一次售后机器人回答“主板保修三年”，但实际某一型号只有一年。根因不是模型太差，而是召回到了旧版本文档。后来我们加了文档版本控制和发布时间过滤，同时要求回答里显示来源文档名，这类问题就下降很多。

## 6. Fine-tuning 和 RAG 应该怎么选？

这题我一般不会回答成“二选一”，而是说它们解决的问题不一样。

RAG 更适合解决“知识更新频繁、需要外部事实依据”的问题，比如客服问答、产品说明、内部制度查询。  
Fine-tuning 更适合解决“输出风格、格式、任务习惯”的问题，比如统一客服话术、固定 JSON 输出、行业措辞适配。

生产上常见的组合其实是“RAG + 指令微调”或者“RAG + 强 prompt 约束”。比如做企业客服时，知识内容来自 RAG，但回答风格希望像资深客服一样自然、礼貌、稳定，这时候可以通过微调或规则模板来统一语气。

## 7. 你怎么评估一个 RAG 系统做得好不好？

我通常会把评估拆成两段：检索评估和生成评估。

检索评估看的是“该不该找到的内容有没有找到”，常见指标有 recall、hit rate、MRR。  
生成评估看的是“回答有没有基于事实、有没有答到点上、有没有胡说”，常见维度有正确性、完整性、可引用性、拒答准确率。

实际项目里，我不会只看离线指标，还会结合真实业务数据。比如客服机器人上线后，我们会看：

- 首轮解决率有没有提升
- 人工转接率有没有下降
- 用户追问率有没有下降
- 高风险问题是否仍然需要人工兜底

举个例子，一个售后机器人离线准确率看着不错，但线上追问率很高。后来发现不是答案错，而是答案太书面，用户看不懂。所以最终评估一定要结合真实交互结果，而不是只看模型分数。

## 8. 为什么多知识库场景要做知识库选择，而不是把所有数据混在一起？

因为数据一旦全混在一起，召回噪音会很高，而且权限边界也容易乱。多知识库的意义，不只是方便管理，更重要的是让检索范围更可控。

我一般会这样解释：用户当前是在问“售后政策”，还是“设备安装规范”，还是“HR 请假制度”？如果提前缩小检索范围，召回结果会更稳，生成成本也更低。

比如一个企业内部助手，售前、售后、研发、行政四类文档全塞一个 collection，用户问“版本升级要不要停机”时，可能同时召回到产品宣传文案和内部技术 SOP，最后答案就容易混。生产里更合理的做法是支持多知识库勾选，或者让 Agent 先判断该查哪一类知识。

## 9. 你怎么做 Prompt 设计，避免提示词越写越长？

我会尽量把 prompt 分层，而不是把所有规则堆到一个大段里。一般至少分成三部分：系统角色、业务约束、输出格式。

真正的经验是，prompt 不是越长越好，而是越清晰越好。很多项目后期 prompt 失控，根因是每出一个线上问题，就往里面加一句规则，最后变成谁都不敢动的大杂烩。

我在生产里更偏向把稳定规则前置到系统 prompt，把场景变量放在运行时消息里，把一些 hard rule 放到程序逻辑里，不全靠模型记住。比如“命中不到知识库时必须明确说明”这种，就应该是稳定约束；而“当前启用的是售后知识库还是安装知识库”这种，是运行时上下文。

## 10. 为什么客服场景经常需要流式输出？

因为用户对“系统有没有在工作”特别敏感。流式输出最大的价值不只是更快，而是让等待过程有反馈，用户会感觉系统是活的。

在智能客服里，这点非常实际。尤其是问题比较复杂、要先检索再生成时，如果页面 5 秒都没反应，用户很容易以为卡住了。流式输出至少可以先把思考过程、工具调用状态、答案前几个字展示出来。

生产里我做过一个设备售后助手，用户问“报修后多久会有工程师联系我”，如果系统整段生成完再显示，主观等待会明显更长。后来改成“先显示正在检索知识库，再流式出答案”，虽然真实总耗时只少了一点，但用户满意度会提升很多。

## 11. Agent 和普通 RAG 问答的区别是什么？

普通 RAG 更像固定流程：检索，然后回答。  
Agent 则是让系统具备“先判断，再选工具，再执行，再继续”的能力。

我一般会说，Agent 适合流程更复杂的任务，比如不仅要查知识库，还要查订单状态、调用工单系统、判断是否要转人工、最后再组织回复。这个时候它就不是单纯回答问题，而是在完成一个业务任务。

比如用户说“我昨天买的空调报码 E3，已经试过重启还是不行，帮我看看还要做什么”。普通 RAG 可能只能把 E3 故障说明查出来；但 Agent 可以先查故障码知识库，再看订单是否在保，再判断是否满足上门条件，最后输出下一步操作建议。

## 12. 生产环境里怎么做模型路由？

模型路由本质上是在效果、成本、速度之间做平衡。不是所有问题都值得上最贵的模型。

我会按任务复杂度分层：

- 简单 FAQ 用轻量模型
- 复杂总结或多轮推理用更强模型
- 高风险问题走更严格流程，必要时直接转人工

举个实际例子，客服系统里“运费谁承担”“退款多久到账”这种问题，轻量模型配合高质量检索就够了；但如果用户粘贴一大段投诉记录，让系统总结问题并生成安抚回复，这种更适合能力更强的模型。这样做能把整体成本打下来，而且不影响核心体验。

## 13. 如何控制大模型应用的成本？

我一般会从四个方向说：

第一，减少无效上下文，别把不相关 chunk 一股脑塞进去。  
第二，做模型分层，不同问题走不同模型。  
第三，缓存高频问题结果。  
第四，优化提示词和输出长度。

生产里最常见的浪费，其实不是模型单价高，而是上下文滥用。比如用户只问一句“支持开发票吗”，结果你把 8 个长 chunk 全塞进去，再让模型输出一大段官话，这个成本是很容易失控的。

我们之前做一个内部知识助手，优化前平均每轮上下文几千 token，后来通过限制 top-k、摘要历史消息、按需引用知识片段，整体成本能明显下降。

## 14. 你怎么做会话记忆，不让历史消息越聊越长？

会话记忆不能等于“把全部聊天记录原样拼进去”。那样上下文会越来越长，成本和噪音都会上升。

我一般会把记忆分成两类：短期上下文和长期关键信息。短期上下文保留最近几轮对话，保证当前语义连续；长期信息则抽取成结构化摘要，比如用户设备型号、地区、订单号、投诉焦点。

比如售后助手里，用户前面说过“我是广东的，机器是 KX-380，问题是制冷差”，后面又追问“那我现在要不要报修”。这时候真正有价值的是那几个关键信息，不是把前面每一句原文都塞回去。

## 15. 你怎么处理知识库更新和旧内容失效问题？

这个问题很关键，因为很多线上事故不是模型错，而是知识库过期。

生产上我会做几件事：

- 文档有版本号
- 文档有更新时间
- 支持重建索引或局部更新
- 检索时尽量保留来源信息
- 对高风险知识做发布审核

举个例子，退款政策从“7 天无理由”调整成“指定品类 15 天无理由”，如果旧文档没下线，机器人就可能一会儿答 7 天，一会儿答 15 天。真正稳定的方案不是靠 prompt 告诉模型“以新为准”，而是保证旧数据别被检索到，或者至少有版本优先级。

## 16. 你怎么做高可用和降级？

大模型应用上线后，不能默认外部模型服务永远稳定。实际生产里，超时、限流、偶发失败都很常见。

我会把降级策略设计在系统里，而不是等出事再补：

- 模型调用失败时重试
- 主模型失败时切备用模型
- 检索失败时给明确提示，不硬答
- 高风险场景直接转人工
- 页面侧做超时和友好提示

比如客服高峰期，某个模型平台响应变慢，如果没有降级，用户看到的就是一直转圈。更稳妥的做法是：先回一个“正在处理中”，超时后切轻量模型或输出保守回复，再把问题挂到人工队列。

## 17. 你怎么做数据安全和权限隔离？

企业场景里，这个问题很现实。不是所有知识都应该被所有人检索到，尤其涉及客户隐私、内部价格、合同条款、研发资料的时候。

我一般会从三层讲：

第一层是知识库级别隔离，不同业务线、角色、租户分开存。  
第二层是检索时做元数据过滤，比如部门、租户、文档级权限。  
第三层是日志脱敏，不把敏感输入原样暴露在日志和监控里。

比如做多租户客服系统时，A 客户的售后文档绝对不能被 B 客户的问题检索出来。这个问题不是模型自己能理解的，而是系统架构必须保证。

## 18. 你怎么看 AI 应用里的“最后一公里”问题？

我理解的“最后一公里”，其实就是 demo 很容易，稳定上线很难。很多团队卡的不是模型效果，而是工程化。

真正难的通常是这些问题：

- 文档怎么持续更新
- 错误答案怎么追踪
- 用户投诉怎么回放
- 模型和知识库怎么灰度
- 线上异常怎么降级

举个例子，一个 RAG demo 可能 2 天就能做出来，但真到了客服场景，你要考虑知识更新流程、引用展示、低置信度转人工、调用链日志、敏感词拦截。能把这些都补齐，才算真正可用。

## 19. 如果用户问题在知识库里找不到，你会怎么回答？

我不会让模型硬猜。我会让系统明确承认当前知识不足，然后尽可能把下一步说清楚。

口语化一点说就是：不知道不可怕，乱答才可怕。尤其客服场景里，错误承诺比拒答更危险。

生产中我常见的回复策略是：

- 先说明当前知识库没有足够信息
- 如果有相近信息，说明只可作为参考
- 给出下一步建议，比如补充订单号、设备型号，或者转人工

比如用户问“海外保修政策在巴西适不适用”，而知识库只有国内政策，这时候最合理的答法不是猜，而是明确说当前知识库没有覆盖该地区规则，建议转人工或查询海外售后专线。

## 20. 你会怎么向非技术业务方解释 RAG 系统上线后的价值？

我一般不会讲太多模型术语，而是讲业务结果。

比如我会说，这套系统上线后，不是为了取代人工客服，而是先把高频、标准、重复的问题拦下来，让人工把时间放在复杂问题上。它的价值通常体现在：

- 缩短响应时间
- 提高首轮解决率
- 降低人工重复劳动
- 让知识沉淀真正能被用起来

如果用实际生产环境举例，一个售后团队每天要处理大量“保修多久”“怎么申请维修”“故障码是什么意思”这类重复问题。RAG 系统上线后，这些标准问题可以先由机器人回答，人工重点处理投诉、升级、异常订单。业务方更容易理解的，不是“我们用了向量检索和大模型”，而是“客服效率上去了，回答也更统一了”。
